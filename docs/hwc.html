HWC — HeadWater Component Syntax

A tiny, convention-friendly format for reusable components built from HeadWater utility classes. Write component definitions in `.hwc` files, compile to plain CSS, then drop into your project.

What is HWC?

HWC (HeadWater Classes) is a small, human-readable component language that maps component names to sets of utility classes. A single `.hwc` file can declare mixins and selectors; the HWC compiler expands them into real CSS using HeadWater's base utilities.

  • Component-first: declare .btn, .card, or utility mixes once and reuse everywhere.
  • Build-time: the compiler looks up utilities in your HeadWater base CSS and emits minimal, optimized CSS.
  • Framework-agnostic: output is plain CSS usable in HTML, React, Vue, SSGs, etc.

Quick Syntax & Examples

Basic mapping: selector = list-of-classes;

.btn = p-2 rounded-full bg-primary text-white;

Hover / pseudo selectors:

.btn:hover = bg-primary-700;      /* class applied on hover */

Mixins: define reusable groups with $name

$smallSpacing = m-2 p-2;
.card = $smallSpacing bg-white rounded shadow-sm;

Full example (.hwc):

// supreme.hwc
$small = m-2 p-2

.btn = bg-primary p-2 rounded-full m-2 text-white
.btn:hover = bg-primary-700
.input = p-2 border rounded focus:ring-2 focus:ring-primary
#spaced-div = $small bg-primary-100

The HWC compiler expands mixins and looks up every utility (like p-2, bg-primary) in your base HeadWater CSS, then emits a minimal .css containing only the used properties.

How to compile

Simple CLI usage (when hw tool is installed):


# Compile supreme.hwc using default base CSS (from HeadWaterGen)
hw compile supreme.hwc -o supreme.css

# Provide a custom base CSS file
hw compile supreme.hwc --base path/to/hw-base.css -o supreme.css

# If you want to scan a project and build only used utilities:
hw scan ./project -o final.css

Result: supreme.css — a small stylesheet containing only the compiled selectors and properties required by your HWC file.

Why use HWC?

  • Collaboration: frontenders and designers can agree on component names (.btn, .card) instead of copying long utility lists.
  • Auditability: compiler checks utilities against base CSS — safe, traceable mapping.
  • Performance: compile-time reduction means smaller CSS in production.
  • Portability: components remain plain HTML + classes — move them across frameworks easily.

Recommended Patterns

Use mixins for shared groups

When many components share the same utility set, define a mixin and reuse it.

$btn-base = p-2 rounded inline-flex items-center;
.btn-primary = $btn-base bg-primary text-white;
.btn-outline = $btn-base border text-dark;

Keep selectors explicit

HWC supports both class selectors and id selectors. Use `.btn` and `#panel` to keep intent clear.

.panel = p-4 border rounded bg-white;

Integration tips

  • Version your compiled CSS: commit compiled component CSS in a `components/` folder if you want reproducible deployments.
  • Use presets: keep `hwconf-*.json` presets for Lite/Basic/Extra/Max builds and compile with the same base CSS.
  • Keep mixins small: mixins are best used for tiny groups (spacing, layout). Don’t replicate huge utility strings inside a mixin.

FAQ (short)

Can HWC reference custom utilities?
Yes. HWC resolves any utility class present in the base HeadWater CSS. If a class doesn't exist, the compiler will warn or error depending on your strict mode.
Is HWC tied to any JS framework?
No. Output is plain CSS. Consume it in vanilla HTML, React, Vue, Svelte, whatever.
Can mixins be recursive?
The compiler detects circular mixin references and reports an error. Keep mixins acyclic.
